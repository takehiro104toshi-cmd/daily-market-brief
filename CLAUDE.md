# daily-market-brief 保守・改善運用プロトコル

このリポジトリは長期運用プロジェクトである。機能追加・改善・修正を行う際は、
以下の更新ポリシー（最上位ルール）を毎回厳守すること。

## 更新ポリシー（最上位ルール・必ず遵守）

1. リポジトリ全体を書き換えない
2. 必要最小限の変更のみ行う
3. 変更ファイルだけ提出する
4. ZIPは依頼された場合のみ作成する
5. 既存コードは極力変更しない
6. 機能追加は既存処理へ影響を与えない形で実装する
7. 1機能=1ファイルを原則とする
8. config値はconfig.yamlへ追加する
9. pytestを必ず実行する
10. CHANGELOG.mdへ更新内容を追記する
11. CLAUDE.md（本ファイル）は運用ルールとして常に遵守する
12. フォルダ構成・Git構成の変更は禁止
13. リネーム・移動は依頼された場合のみ実施
14. 毎回リポジトリ構成を推測しない（既知の構成・ドキュメントに基づいて作業する）
15. GitHub Pages・GitHub Actions・Secretsは変更しない（依頼がある場合のみ）

## 設計原則（設計より実装を優先しない）

このプロジェクトでは実装よりも設計の一貫性を優先する。

以下はChatGPT（監督者）が決定する事項であり、Claude Codeは独自判断で
変更してはならない。

- ディレクトリ構成
- システム設計
- ファイル名
- 設定ファイル構成
- ワークフロー
- GitHub Actions
- GitHub Pages
- ニュース評価ロジック
- スコアリングロジック
- 営業思想
- ストラテジスト思想
- UIデザイン
- 出力フォーマット

改善案がある場合は、**変更せず提案のみ**行うこと（ユーザーの承認・明示的な
依頼があって初めて変更する）。

## リファクタリング

ユーザーから依頼されない限り、以下は禁止。

- リネーム
- ファイル移動
- 関数統合
- 大規模整理
- 不要コード削除

技術的に望ましく見えても、既存の安定動作を優先する。

以下は、上記ポリシーを実務でどう満たすかの詳細（1〜11の補足）。

## 更新方針

機能追加・改善を行う場合、以下を最優先する。

- 既存機能を壊さない
- 既存APIを変更しない（デフォルト値付きで拡張し、既存の呼び出し箇所に影響を与えない）
- 後方互換性を維持する
- 小さい差分で実装する

## 実装後の出力ルール

実装が終わったら、必ず以下を順に出力する。

### ①変更概要

例:
```
v1.09
追加
・営業ポイント自動生成
改善
・重要度スコア調整
修正
・HTML表示崩れ修正
```

### ②変更ファイル一覧

変更したファイルをすべて列挙する。新規ファイルには`【新規】`、削除ファイルには`【削除】`を付ける。

### ③差し替え方法

「既存ファイルを `<path>` のみ置き換えてください」のように、ユーザーがどのファイルだけを
差し替えればよいかを明記する。

### ④Git情報

`git status` / `git diff --stat` / `git log -1` の結果を表示する。

## ZIP作成ルール

ZIPは**ユーザーから依頼された場合のみ**作成する。基本は常に**変更ファイルだけ**を
`SendUserFile` で送る。リポジトリ全体は送らない。変更ファイル数が多い場合でも、
勝手にZIPへ切り替えず、変更ファイルを個別に列挙して送る。

## ディレクトリ構成

新しい分析機能は可能な限り `src/analysis/` 配下へ、1機能＝1ファイルで追加する。
既存ファイルの巨大化は避ける。

## config

設定値は `config.yaml` へ追加する。Pythonコードへ直接埋め込まない。

## テスト

変更後は必ず `pytest` を実行し、結果（例: `89 passed`）を報告に含める。

## Git コミットメッセージ

`Add ○○` / `Improve ○○` / `Fix ○○` の形式にする。

## 更新履歴

リポジトリ直下の `CHANGELOG.md` を必ず維持し、更新のたびに
`## v1.09` のような見出しの下に「追加／改善／修正」を追記する。

## 最終報告フォーマット

作業完了時は、必ず以下の形式で締めくくる。

```

```
